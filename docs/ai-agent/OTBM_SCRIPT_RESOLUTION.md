# OTBM script-resolution audit

`tools/ai-agent/otbm_script_resolution_tool.py` correlates map-level OTBM mechanics with the active Canary Lua and XML registrations that can execute them.

The tool is read-only. It does not edit the map, scripts, XML files, or runtime configuration.

## Why this audit exists

An OTBM item may contain an `actionId`, `uniqueId`, teleport destination, or house-door identifier. Inventorying those attributes is not enough: before a map region is copied, removed, or reconstructed, the project needs to know whether Canary has a matching handler and how that handler is reached.

The resolver separates two questions:

1. **Runtime resolution** — is there evidence that Canary executes this mechanic?
2. **Review disposition** — if no handler was found, has the identifier been explicitly classified for preservation and later gameplay review?

A reviewed unresolved identifier remains labelled `unresolved`; review rules never falsely convert it to `handled`.

## Inputs

The first positional argument is the report generated by `otbm_item_audit_tool.py`. It must contain `mechanicPlacements` and `mapMechanicItems`.

By default, only the active datapack roots are scanned:

```text
data
data-otservbr-global
```

Alternative datapacks such as `data-canary` are not mixed into the result unless passed explicitly with `--script-root`. This avoids duplicate handlers and false conflicts.

## Run the resolver

```bash
python tools/ai-agent/otbm_script_resolution_tool.py \
  artifacts/OTBM_ITEM_AUDIT.json \
  --repository-root . \
  --script-root data \
  --script-root data-otservbr-global \
  --review-rules docs/ai-agent/OTBM_SCRIPT_REVIEW_RULES.json \
  --output artifacts/OTBM_SCRIPT_RESOLUTION.json
```

Useful options:

- `--rules` adds explicit runtime registrations that cannot be expressed statically;
- `--review-rules` applies human dispositions to unresolved identifiers;
- `--strict-runtime` fails while any runtime identifier remains unresolved, even when reviewed;
- `--allow-findings` writes the report and returns zero regardless of findings.

## Supported registration forms

The static resolver recognizes:

- RevScript `Action()` and `MoveEvent()` constructors;
- direct `:aid(...)`, `:uid(...)`, `:id(...)`, and `:position(...)` registrations;
- numeric loops and ranges;
- `pairs()` / `ipairs()` loops over statically defined tables;
- XML action and movement registrations;
- target-side comparisons such as `target.actionid == 1234` and `target.actionid ~= 1234`;
- target table lookups such as `config[target.actionid]` and `boss[target.uid]`;
- bounded target ranges such as `target.actionid >= 4208 and target.actionid <= 4223`;
- generic item-id handling for doors and keys defined by `items.xml`;
- quest chest fallback from `uniqueId` to a registered `actionId`, including `actionId 2000`;
- engine-handled teleport destinations and house-door identifiers.

MoveEvent types are independent. `stepin`, `stepout`, `additem`, `removeitem`, `equip`, and `deequip` registrations for the same identifier do not conflict with each other.

## Runtime statuses

Common placement and identifier statuses include:

- `handled-directly`
- `handled-by-range`
- `handled-generically`
- `handled-by-item-id`
- `handled-by-action-id`
- `handled-by-unique-id`
- `handled-by-position`
- `handled-as-target`
- `handled-by-fallback`
- `handled-by-engine`
- `handled-multiple`
- `partially-resolved`
- `referenced-only`
- `unresolved`
- `conflicting`

A conflict is reported only when multiple distinct handlers compete for the same event type. A `stepin` and `stepout` pair is not a conflict.

## Review rules

The versioned review file uses the format:

```json
{
  "format": "canary-otbm-script-review-rules-v1",
  "rules": [
    {
      "namespace": "actionId",
      "values": [12345],
      "disposition": "needs-manual-review",
      "reason": "No active runtime handler was confirmed; preserve the map attribute."
    }
  ]
}
```

Allowed dispositions:

- `intentional-marker`
- `legacy-unused`
- `missing-script`
- `needs-manual-review`
- `preserve-until-reviewed`

Review rules should use explicit values or narrow, justified ranges. They are safety records, not runtime registrations.

## Report success

The default report has `ok: true` when:

- there are no conflicting placements; and
- every runtime-unresolved identifier has an explicit review disposition.

This does **not** mean every identifier has a runtime handler. Use `--strict-runtime` when zero runtime-unresolved identifiers is required.

The report contract is `docs/ai-agent/OTBM_SCRIPT_RESOLUTION_REPORT.schema.json`.

## Real-map validation

The implementation was validated against the uploaded `otservbr(2).otbm`, the item audit from client 15.25 data, and the active `data` plus `data-otservbr-global` scripts.

One validation run produced:

- 9,339 map-mechanic placements;
- 8,964 runtime-resolved placements;
- 375 partially or fully unresolved placements;
- zero conflicting placements;
- 1,284 distinct action/unique identifiers;
- 151 runtime-unresolved identifiers, all explicitly classified as `needs-manual-review`;
- zero unreviewed identifiers.

These counts describe that specific map and repository snapshot; they are not hard-coded expectations.

## Limitations

Static analysis cannot safely evaluate arbitrary Lua execution, dynamically loaded tables, generated identifiers, database state, or runtime configuration. Such expressions are retained in `unresolvedDynamicRegistrations`.

The resolver deliberately prefers an explicit unresolved finding over guessing. A map attribute must not be deleted or remapped solely because static analysis did not find a handler.

## Tests

```bash
python -m unittest tools/ai-agent/test_otbm_script_resolution.py -v
```

The tests cover direct IDs, numeric ranges, table loops, target-side tables and ranges, distinct MoveEvent types, real conflicts, quest fallback, door/key handling, engine mechanics, active datapack selection, dynamic-registration warnings, and review dispositions.
