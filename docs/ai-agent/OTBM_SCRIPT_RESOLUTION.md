# OTBM script-resolution audit

This tool resolves map-level `actionId` and `uniqueId` placements from a raw
OTBM item scan to active Canary Lua/XML handlers. It is read-only and does not
modify the map, datapack, client assets, or server configuration.

## Inputs

1. A Canary repository checkout.
2. A `canary-otbm-item-scan-v1` JSON file generated by
   `tools/ai-agent/otbm_item_audit_scan.cpp`.

Only active datapacks are scanned by default:

- `data`
- `data-otservbr-global`

`data-canary` is intentionally excluded unless explicitly passed with another
`--datapack` option.

## Generate the raw map scan

```sh
g++ -std=c++20 -O2 tools/ai-agent/otbm_item_audit_scan.cpp \
  -o artifacts/otbm_item_audit_scan

artifacts/otbm_item_audit_scan \
  /path/to/otservbr.otbm \
  artifacts/OTBM_ITEM_SCAN.json
```

## Resolve handlers

```sh
python tools/ai-agent/otbm_script_resolution_tool.py \
  --root . \
  --item-scan artifacts/OTBM_ITEM_SCAN.json \
  --output artifacts/OTBM_SCRIPT_RESOLUTION.json \
  --map /path/to/otservbr.otbm \
  --expected-map-sha256 a80de1dda6a9aca3956a9d5b7fb2e0caebb451570d26853fc21beb40d5f31da2 \
  --allow-unresolved
```

Without `--allow-unresolved`, the CLI returns exit code `2` while any identifier
is unresolved or partially resolved. Dispatch conflicts always make the report
fail.

The output follows
`docs/ai-agent/OTBM_SCRIPT_RESOLUTION_REPORT.schema.json`.

## Recognized registration patterns

The static evaluator recognizes:

- `Action()` and `MoveEvent()` objects;
- helper-backed `BossLever(...)` action objects;
- direct `:aid(...)`, `:uid(...)`, `:id(...)`, and `:position(...)` calls;
- numeric ranges and comma-separated lists;
- numeric `for` loops;
- `pairs()` and `ipairs()` over static tables;
- shared door tables loaded from `data/libs/tables`;
- static `table.insert(...)` collection building;
- legacy XML `<action>` and `<moveevent>` ranges;
- `item.actionid`, `target.actionid`, `item.uid`, `target.uid`, and getter forms;
- item-ID fallback dispatch;
- generic quest-chest action ID `2000`;
- engine-handled teleport destinations and house-door IDs.

Dynamic expressions that cannot be evaluated safely are retained in
`dynamicRegistrations`. The resolver never invents values and never hides an
unknown expression as handled.

## Status meanings

- `handled-directly` â€” exact static registration;
- `handled-by-range` â€” registration came from a range/loop;
- `handled-generically` â€” known generic mechanism such as quest chest AID 2000;
- `handled-by-item-id` â€” dispatch falls back to the placed item's ID;
- `handled-as-target` â€” an active handler explicitly inspects the target/item ID;
- `handled-by-engine` â€” native map mechanic such as teleport or house door;
- `handled-multiple` â€” more than one non-conflicting handler category applies;
- `unresolved` â€” no safe static resolution was found.

A conflict is reported only when separate registrations compete in the same
namespace, handler kind, and event type. A `stepin`/`stepout` pair is not a
conflict. Handlers reached through different item-ID fallbacks on different map
placements are also not conflicts.

## Safety and limitations

- Source files are decoded with replacement for invalid UTF-8 but never changed.
- Lua is not executed.
- The parser deliberately supports a bounded, deterministic subset of Lua.
- Runtime-generated IDs, database-derived values, and deeply dynamic helper
  metaprogramming remain explicit warnings.
- A resolved handler proves that dispatch can reach code; it does not prove that
  the quest or mechanic behaves exactly like Real Tibia.
- Review unresolved IDs with their item IDs and world coordinates before adding,
  deleting, or changing any map attribute.
